Python Data Structures Interview Guide

When to use what — with concrete examples for each question

Storage & Lookup

Choosing how to store data based on how you'll retrieve it.

Do I need fast lookup by a meaningful key?
dict O(1) avg ★★★

Why dict?

A dict stores key→value pairs with a hash table under the hood. Any lookup by key costs O(1) on average — no matter how many entries. Use it whenever you access data by a meaningful label rather than a position.

Example

# Map student names to grades
grades = {
  "Alice": 92,
  "Bob": 85,
  "Clara": 78,
}

# O(1) lookup — instant, no loop
print(grades["Alice"])  # 92

# Safe access with default
grades.get("Dan", 0)   # 0 (not KeyError)
Do I need to check if a value exists fast?
set O(1) avg ★★★

Why set?

Sets use hashing just like dicts — checking membership with in is O(1). If you only need to know "is X in my collection?" and don't need a value or position, a set is the right tool.

Example

# Banned usernames — checked on every login
banned = {"admin", "root", "superuser"}

username = "root"

if username in banned:   # O(1) !
  print("Access denied")
else:
  print("Welcome")
Do I need ordered access by position (index)?
list O(1) by index ★★

Why list?

Lists are sequences — elements have a position and you access them by index in O(1). Great when order matters and you'll iterate, slice, or access by number. But searching by value is O(n).

Example

# Steps of a process — order matters
steps = ["parse", "validate", "save", "notify"]

# Access by index — O(1)
print(steps[0])    # "parse"
print(steps[-1])   # "notify"

# Slice — get first two steps
print(steps[:2])   # ["parse", "validate"]
Do I need fixed, immutable data (e.g. coordinates, config)?
tuple O(1) by index ★★

Why tuple?

Tuples are like lists that can never change. Use them to signal "this data belongs together and must not be altered." They're also hashable, which makes them usable as dict keys — lists cannot be.

Example

# GPS coordinates — should never change
paris = (48.8566, 2.3522)
london = (51.5074, -0.1278)

# Unpack naturally
lat, lon = paris
print(lat)  # 48.8566

# Use as dict key (lists can't!)
city_map = {paris: "Paris", london: "London"}

Counting & Grouping

Using dicts to aggregate, tally, and classify data.

Do I need to count occurrences of elements?
dict / Counter O(n) build, O(1) lookup ★★★

Why dict or Counter?

Build a frequency map in one pass with a dict. collections.Counter does the same automatically and adds useful methods like most_common(). This pattern is in almost every coding test.

Example

from collections import Counter

words = ["apple", "banana", "apple", "cherry", "banana", "apple"]

# Manual dict approach
counts = {}
for w in words:
  counts[w] = counts.get(w, 0) + 1
# {"apple": 3, "banana": 2, "cherry": 1}

# Counter shortcut
c = Counter(words)
c.most_common(2)  # [("apple",3), ("banana",2)]
Do I need to group items by category?
dict of lists O(n) ★★★

Why dict of lists?

Map each category key to a list of matching items. Classic interview problem: "group anagrams together" — the key is the sorted word, the value is the list of anagrams. defaultdict(list) removes the need to check if the key exists first.

Example

from collections import defaultdict

words = ["eat", "tea", "tan", "ate", "nat", "bat"]

groups = defaultdict(list)

for w in words:
  key = tuple(sorted(w))  # ("a","e","t")
  groups[key].append(w)

# {('a','e','t'): ['eat','tea','ate'],
#  ('a','n','t'): ['tan','nat'],
#  ('a','b','t'): ['bat']}
Do I need to track the first/last seen item?
dict O(1) per item ★★

Why dict?

Store each element's index as you iterate. When you see it again, the stored index is where it was first (or last) seen. This powers classic problems like "first non-repeating character" or "two sum".

Example

# First non-repeating character
s = "leetcode"

seen = {}
for i, ch in enumerate(s):
  if ch not in seen:
    seen[ch] = i        # first index
  else:
    seen[ch] = -1       # mark as duplicate

result = min(
  (i for i in seen.values() if i != -1)
)
print(s[result])  # "l"

Uniqueness & Sets

Leveraging set operations for deduplication and intersection problems.

Do I need to remove duplicates?
set O(n) ★★★

Why set?

Converting to a set drops all duplicates in one operation. Convert back to a list if you need indexing. Warning: order is not guaranteed — if order matters, use dict.fromkeys() instead (preserves insertion order).

Example

nums = [3, 1, 4, 1, 5, 9, 2, 6, 5, 3]

# Fast — order not preserved
unique = list(set(nums))

# Preserves insertion order (Python 3.7+)
unique_ordered = list(dict.fromkeys(nums))
# [3, 1, 4, 5, 9, 2, 6]

# Count unique elements
print(len(set(nums)))  # 7
Do I need intersection, union, or difference of two groups?
set O(min(a,b)) ★★★

Why set?

Sets have built-in operators for all set algebra. These are extremely readable and efficient. Master the 4 operators: | union, & intersection, - difference, ^ symmetric difference.

Example

python_devs = {"Alice", "Bob", "Clara"}
js_devs     = {"Bob", "Dan", "Clara"}

# Both Python AND JS
python_devs & js_devs    # {"Bob", "Clara"}

# Python OR JS (or both)
python_devs | js_devs    # {"Alice","Bob","Clara","Dan"}

# Python only (not JS)
python_devs - js_devs    # {"Alice"}

# One but not the other
python_devs ^ js_devs    # {"Alice", "Dan"}
Do I need to find common elements between two lists?
set a & b O(min(a,b)) ★★★

Why set intersection?

A nested loop to find common elements is O(n²). Converting both lists to sets and using & is O(n). This is a very common interview optimization — the examiner is often waiting for you to spot it.

Example

list1 = [1, 2, 3, 4, 5]
list2 = [3, 4, 5, 6, 7]

# Naive: O(n²) — DON'T do this
common = [x for x in list1 if x in list2]

# Optimal: O(n) — DO this
common = list(set(list1) & set(list2))
# [3, 4, 5]

# Why? "in list2" is O(n) per element!
# "in set2" would be O(1) per element.

Order & Sequence

When the position or sequence of elements is what matters.

Do I need to maintain insertion order and allow changes?
list O(1) append, O(n) insert ★★

Why list?

Lists preserve insertion order and can grow or shrink freely. Appending to the end is O(1). Inserting or removing from the middle shifts elements and costs O(n) — keep that in mind for large collections.

Example

# Build a result list progressively
results = []

for n in [4, 2, 7, 1, 9]:
  if n % 2 == 0:
    results.append(n)    # O(1)

print(results)  # [4, 2]

# Equivalent with list comprehension
results = [n for n in [4, 2, 7, 1, 9] if n % 2 == 0]
Do I need to use data as a dictionary key?
tuple O(1) hash ★★★

Why tuple?

Dict keys must be hashable. Lists are mutable and cannot be hashed — using one as a key raises a TypeError. Tuples are immutable and hashable, making them ideal as composite keys (e.g. grid coordinates, multi-field lookups).

Example

# 2D grid with values at each cell
grid = {}
grid[(0, 0)] = "start"
grid[(3, 4)] = "treasure"
grid[(5, 5)] = "end"

print(grid[(3, 4)])  # "treasure"

# This would crash — lists aren't hashable
# grid[[3, 4]] = "treasure"  ← TypeError!
Do I need a stack (LIFO — last in, first out)?
list O(1) ★★★

Why list?

A list doubles as a stack: .append() pushes, .pop() pops — both O(1). Classic use cases: balanced parentheses, undo history, DFS traversal. This is one of the most tested patterns in interviews.

Example

# Check balanced parentheses
def is_balanced(s):
  stack = []
  pairs = {")": "(", "]": "[", "}": "{"}
  for ch in s:
    if ch in "([{":
      stack.append(ch)         # push
    elif not stack or stack.pop() != pairs[ch]:
      return False             # pop
  return not stack

is_balanced("([{}])")   # True
is_balanced("([)]")     # False
Do I need a queue (FIFO — first in, first out)?
deque O(1) both ends ★★★

Why deque, not list?

list.pop(0) is O(n) — every element shifts left. A deque (double-ended queue) gives O(1) on both sides. Use it for BFS, sliding window, or any FIFO task.

Example

from collections import deque

# BFS traversal of a graph
def bfs(graph, start):
  visited = set()
  queue = deque([start])
  while queue:
    node = queue.popleft()   # O(1) !
    if node not in visited:
      visited.add(node)
      queue.extend(graph[node])
  return visited

# list.pop(0) would be O(n) — never for queues

Performance Traps

Common mistakes interviewers look for — and how to fix them.

Is x in list okay for large data?
use set instead list O(n) → set O(1) ★★★

The trap

The in operator on a list scans every element until it finds a match — O(n). Inside a loop, this becomes O(n²). Converting to a set before the loop brings lookup down to O(1).

Example

# ❌ O(n²) — "in list" inside a loop
def has_duplicate_slow(nums):
  seen = []
  for n in nums:
    if n in seen:   # O(n) each time!
      return True
    seen.append(n)
  return False

# ✅ O(n) — use a set
def has_duplicate_fast(nums):
  seen = set()
  for n in nums:
    if n in seen:   # O(1) each time
      return True
    seen.add(n)
  return False
Do I need a default value when a key is missing?
.get() / defaultdict O(1) ★★

The trap

Accessing a missing key raises KeyError. Wrapping every access in a try/except is verbose. Use .get(key, default) for one-off access, or defaultdict when every missing key should auto-initialize to the same type.

Example

from collections import defaultdict

# ❌ Verbose — checking key existence manually
counts = {}
for ch in "hello":
  if ch not in counts:
    counts[ch] = 0
  counts[ch] += 1

# ✅ Idiomatic — .get() with default
counts = {}
for ch in "hello":
  counts[ch] = counts.get(ch, 0) + 1

# ✅ Cleaner — defaultdict(int)
counts = defaultdict(int)
for ch in "hello":
  counts[ch] += 1  # starts at 0 automatically
Do I need to insert/delete at the front of a sequence often?
deque not list list O(n) → deque O(1) ★★★

The trap

list.insert(0, x) and list.pop(0) are both O(n) — every element must shift. A deque is designed for O(1) operations at both ends. In a sliding window or BFS context, this can be the difference between O(n) and O(n²).

Example

from collections import deque

# Sliding window maximum (last k elements)
def sliding_max(nums, k):
  dq = deque()   # stores indices
  result = []
  for i, n in enumerate(nums):
    while dq and nums[dq[-1]] < n:
      dq.pop()           # O(1) from right
    dq.append(i)
    if dq[0] == i - k:
      dq.popleft()       # O(1) from left !
    if i >= k - 1:
      result.append(nums[dq[0]])
  return result
Interview tip: When you write a solution, always ask yourself out loud: "What's the complexity of my lookups?" Interviewers award extra points for catching O(n) lookups inside loops and replacing them with O(1) set or dict operations — even if your first solution was correct.
